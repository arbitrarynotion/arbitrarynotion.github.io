Project Summary

Accessible Custom Editor (ACE) UI is a Unity development framework that streamlines custom editor creation and provides easy access to real-time data manipulation and visualization features.

As this framework is a developer tool, a little developer context may be helpful.

First, A Little Context

Developing a game in Unity often involves writing custom game scripts to build a game's intended functionality. The scripts are then added to object's within the Unity editor and displayed in a special editing window referred to as the "Inspector", a part of the Unity interface which displays a selected object's attached scripts as "Components". A custom game script has the option of exposing parts of its data to be displayed as settings in the Inspector, those settings allowing a Unity developer to adjust the behavior of the game script without needing to engage with the script's code.

Project Purpose

When developing a game solo or on a small team, I'm often playing multiple roles. While I try to focus the majority of my effort on programming to keep pushing myself to expand my CS skills, I still do a lot of work as an artist and a game developer. When writing a game script, I have two high-level goals in mind:

  1. Provide the required functionality efficiently with clean and maintainable code.
  2. Ensure that the script's intended behavior and settings can be easily understood and adjusted from within the Unity editor.

As a programmer, it can be easy to unintentionally neglect that second goal.

However, Unity's default approach to drawing script UI settings produces results that can be difficult to visually parse and aren't easily adjusted. While it's possible to improve those results with grouping, indenting, and font style adjustments, it can be prohibitively time-consuming as it requires a significant amount of additional boilerplate-heavy code which can't simply be reused across multiple scripts.

ACE UI addresses these issues by providing a framework that automates boilerplate code, is easily reusable across multiple scripts, and even facilitates real-time Inspector layout modifications.

The Problem this Framework Solves

The pain point this framework addresses is the way custom game script settings are displayed in the Inspector. By default, without writing extensive extra editor code for each game script to manually guide how settings are drawn, the settings get clumped together in a way that can be difficult to visually parse. In particular, there is no easy way to group settings into sections that can be folded (hidden under a heading) or toggled (enabled/disabled as a group). Further, the size and spacing between settings can not be easily changed to accommodate differences in visual comprehension.
